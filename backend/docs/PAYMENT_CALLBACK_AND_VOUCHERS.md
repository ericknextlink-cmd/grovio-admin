# Payment callback fix & Admin vouchers

## Payment callback failing: `orders_payment_method_check`

**Symptom:** After Paystack payment, verify-payment returns 400 and logs:

```text
new row for relation "orders" violates check constraint "orders_payment_method_check"
```

**Cause:** The `orders` table has a CHECK constraint on `payment_method` that did not allow the value `'paystack'` we were inserting.

**Fix (two parts):**

1. **Run this migration in Supabase SQL editor** (once):

   File: `supabase-orders-payment-method-constraint.sql`

   It drops the existing `orders_payment_method_check` and adds a new one that allows: `paystack`, `card`, `mobile_money`, `bank_transfer`, `bank`, `ussd`, `qr`, `eft`, `unknown`.

2. **Backend code** (`order.service.ts`): When creating the order after verification we now set `payment_method` from Paystack’s `channel` when it’s one of the allowed values (e.g. `mobile_money`, `card`); otherwise we use `'paystack'`. So even before the migration, if your DB already allows `card`/`mobile_money`/etc., verification may succeed; the migration ensures `'paystack'` and other channels are always allowed.

After applying the migration and deploying the backend, revisiting the payment callback URL will verify again and create the order (or show the existing order if already created).

## Admin: discount vouchers and “assign to single user”

**Current state:**

- **Database:** Tables exist and are used by the main app:
  - `discount_vouchers` – voucher definition (code, type, value, `image_type`: `regular` | `nss`, validity, etc.)
  - `user_vouchers` – links a user to a voucher (assignments); `used_at` set when applied at checkout
- **Backend (customer API):** Under `/api/vouchers`:
  - `POST /validate` – validate a code and get discount (used at checkout)
  - `GET /` – list vouchers assigned to the current user
- **Voucher images:** `VoucherImageService` (and optional API) can generate images from templates in `SUPABASE_STORAGE_BUCKET/templates/` (e.g. `Voucher Orange template.png`, `Voucher Blue template.png`) with dynamic text (code, name, expiry, etc.).

**What is not implemented yet:**

- **Admin API** for vouchers: no routes under `/api/admin` to:
  - Create a voucher (code, type, value, image_type, validity, etc.)
  - Assign a voucher to a single user (insert into `user_vouchers`) or to “all users” (e.g. no row in `user_vouchers` and validation logic treats it as global, depending on your design).
- **Admin UI:**
  - No admin page that shows “discount images” (preview or list of generated voucher images).
  - No admin page to create vouchers and assign them to a single user (or bulk).

**Where the logic would live:**

- **Assign discount to a single user:** Backend would insert (or upsert) into `user_vouchers` with `user_id` and `voucher_id`. The existing `VoucherService` (used by the customer API) already considers user assignments when validating.
- **Discount images:** Served or generated by `VoucherImageService`; an admin endpoint could return a generated image URL or blob for a given voucher (and optionally user) for preview/list in the admin UI.

To add “assign to single user” and “show discount images” in the admin:

1. Add admin voucher routes (e.g. under `/api/admin/vouchers` or `/api/admin`): create voucher, list vouchers, assign voucher to user(s), and optionally “preview image” for a voucher.
2. Add an admin frontend page (e.g. `/admin/vouchers` or a section under Settings/Discounts) that:
   - Lists vouchers and allows creating new ones (code, type, value, image_type, validity).
   - For each voucher, allows assigning to a single user (and optionally “all users”).
   - Shows or links to the discount image (using the new admin image preview endpoint and `VoucherImageService`).

Schema and validation are already in place in `discount_vouchers` and `user_vouchers`; the missing pieces are the admin API and the admin UI that call them.
